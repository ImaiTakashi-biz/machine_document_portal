# 要件整理

## 確定事項

- プロジェクト名は Machine Document Portal、フォルダ名は `machine_document_portal` とする。
- Googleスプレッドシートの「稼働中工程内検査シート」をWebアプリへ置き換える。
- 号機ごとに現在生産中の品番と品名を表示する。
- 品番はGoogleスプレッドシート「生産中」から取得する。
- 工程内検査シートはSharePoint、加工図はNASに存在する。
- 各号機に2種類の資料ボタンと取得状態を表示する。
- 表示対象は「生産中」シートのD列に存在する号機とし、追加・削除は次回更新時に反映する。
- 号機はグループ、数値の号機番号の順に並べる。
- 当面はPostgreSQLへ保存せず、アプリのプロセス内メモリで状態を保持する。
- SQLiteは使用しない。
- PostgreSQL構成は将来の選択肢として残すが、`PERSISTENCE_MODE=memory` では接続しない。
- UIは `docs/DESIGN` のガイド、ロゴ、ブランド配色を参考にする。サイドバーヘッダーには ARAI ロゴ（`/design-assets/arai_logo.png`）を表示する。
- ブラウザタブの favicon、ホーム画面追加、Web App Manifest（`display: standalone`）には `app/static/icons/` の Machine Document Portal アイコンと `app/static/manifest.json` を使用する。Service Worker は使用せず、インストール後も画面更新・自動再読込は通常のブラウザ表示と同じ挙動とする。
- Windows業務PCと工場内タブレットで利用する。
- 本番ではGoogle Sheets APIへ読み取り専用で接続し、サンプルデータは画面確認用とする。
- 工程内検査シートはMicrosoft Graphのアプリケーション認証でSharePointから読み取り専用で取得する。
- SharePoint連携は `Sites.Selected` と対象サイトの `Read` 権限を使用し、全サイト対象の読み書き権限は使用しない。

## 今回の仮仕様

- 号機グループ、台数、色、サンプル状態は `sample_data/machines.json` で変更できる。
- サンプルデータの既定号機は「生産中・両資料あり」とし、A-1～B-3で全代表状態を確認する。
- 号機カードはグループごとの可変グリッドとし、画面幅に応じて列数を変える。
- 工程内検査シートと加工図は別々の状態を持つ。
- 内部比較用の品番キーはNFKC、大文字化、空白除去、ハイフン統一を行い、枝番・改訂番号・アンダースコアは保持する。ただし、SharePointとNASのファイル照合にはGoogle Sheetsから取得した元の品番文字列を使用する。
- 加工図は `NAS_DRAWING_DIRECTORY` 配下でH列の品番と同名のPDFを検索し、専用の別タブで1ページ目を画像表示する。
- 工程内検査シートはH列の品番とSharePointファイル名の拡張子を除いた文字列を正規化せず照合する。完全一致または`品番-[1-9][0-9]*`を対象とし、大文字・小文字、全角・半角、空白、ダッシュ表記の違いは同一視しない。
- ファイル名が別の稼働中品番へ完全一致する場合は、その完全一致を連番関連付けより優先する。
- 工程内検査シートが1件の場合は直接開き、複数の場合は件数を表示して別タブの選択画面に全ファイルの名前と保存場所を数値順で表示する。
- 13:00の不足資料確認では、完全一致または連番ファイルが1件以上あれば工程内検査シート設定済みと判定する。
- サイドバーから工程内検査シート、出荷検査表、測定機器点検表の共通リンクを別タブで開く。
- 日時はタイムゾーン付きで扱い、UIでは現地時刻の文字列表現を行う。
- UIの最終更新時刻は日本時間（Asia/Tokyo）で表示する。
- 生産状態はA列から取得し、稼働中／停止中／生産終了／セット中をそれぞれ緑／赤／紫／青で表示する。
- 号機、品番、資料リンク、取得状態はプロセス内メモリで使用し、最新画面だけを再起動復元用JSONへ保存する。
- 外部検索エラー時は前回の正常リンクを再利用しない。
- メモリ共有を行わないため、当面のUvicorn起動は1ワーカーとする。
- アプリ起動時と手動更新時はGoogle Sheets、SharePoint、NASを完全同期する。
- `AUTO_REFRESH_SECONDS` 秒ごとにGoogle Sheetsを取得し、前回から品番が新規・変更された号機だけSharePointとNASを再検索する。A列・I列だけの変更では資料を再検索しない。`0`で無効にする。
- `DOCUMENT_REFRESH_TIMES` に指定した日本時間に、Google Sheetsの「生産中」シートを再取得して画面へ反映し、その時点の全号機についてSharePointとNASも完全更新する。複数時刻はカンマ区切り、空欄で無効にする。
- `AUTO_REFRESH_SECONDS` または `DOCUMENT_REFRESH_TIMES` の同期が正常完了した後、各端末で開かれている画面は更新完了を検知して自動再読込する。利用者による更新・印刷操作中は再読込を保留する。
- 「最新情報に更新」ボタンは必要以上に横へ広げず、最終更新日時を右側、工程内検査シート・加工図面を更新するときの操作説明を下側へ1行で表示する。
- サイドバーには内部保存方式や内部状態点を表示せず、利用者が把握すべき外部連携エラーは画面上部の通知と資料ボタンで表示する。
- 毎日13:00に翌営業日シートの `B36:K36` と `B40:K40` を列ごとに確認する。B36:K36の日付が対象日より前の列は、同じ列のB40:K40の品番を処理対象から除外する。対象日と同じ日付または対象日より後の日付の列は処理を続行する。
- 対象となった品番についてSharePointとNASの不足資料をARAICHATルームID 24へ送信する。
- 毎日14:30に13:00と同じシートの `B36:K36` と `B40:K40` を再取得し、同じ日付条件でSharePointとNASを再確認する。その時点でも確認できない品番がある場合だけ、現在の不足内容をARAICHATルームID 24へ送信する。
- 13:00通知の見出しは「【翌営業日セット予定分の検査シート・加工図面確認通知】」、14:30通知の見出しは「【翌営業日セット予定分の再確認（14:30）】」とする。
- 13:00と14:30の通知には、品番ごとの必要な対応、工程内検査シートのSharePoint保存場所、加工図面のNAS保存場所を表示する。資料確認用のサイドバー案内は表示しない。
- 毎日15:00に13:00と同じシートの `B36:K36` と `B40:K40` を再取得し、同じ日付条件で対象となった品番に一致するNAS加工図を指定プリンターへ1品番1部印刷する。
- 同じ翌営業日シートの13:00通知、14:30通知、印刷はそれぞれ繰り返さず、部分失敗時は未完了項目だけ再処理する。
- 15:00時点で加工図が存在しない品番は自動再検索・自動印刷の対象外とし、アップロード後に手動発行する。
- 15:00時点で存在したPDFをプリンターへ送信できなかった場合は、3分後、5分後、10分後に未印刷品番だけを自動再実行する。
- 自動再実行後も未完了の場合は、通常時には存在しない「印刷の確認」を左サイドバーへ表示し、利用者が専用画面から未印刷分だけを印刷できるようにする。
- 利用者による印刷もアプリを起動しているWindows PCから指定プリンターへ送信し、操作端末のプリンターは使用しない。
- 送信結果が不明な品番は自動再印刷せず、利用者による用紙確認後に印刷済みまたは再印刷を選択する。
- プリンターへ送信する前に品番を結果確認が必要な状態として保存し、状態保存に失敗した場合はプリンターへ送信しない。送信後に処理が中断しても自動再印刷せず、利用者の用紙確認を待つ。
- 定時処理状態ファイルが破損、不正形式、読み書き不能の場合は空の状態として再開せず、13:00・14:30の通知、自動印刷、手動再印刷を安全停止する。通常のダッシュボード表示は継続する。
- 印刷処理履歴と技術情報は利用者画面に表示せず、開発者向けログだけに記録する。
- 本番切替時は従来の印刷スクリプトを停止して二重実行を防ぎ、外部サービスとプリンターの確認後、13:00より前に定時処理を有効化する。
